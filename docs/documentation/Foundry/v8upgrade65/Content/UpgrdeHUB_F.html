Upgrade from HCL Server (middleware) to latest version of Foundry (now, HCL Foundry)

The following sections help you upgrade middleware/server from 6.5.x to Foundry 7.x:

  1. Software and Hardware Requirements
  2. Approaches to upgrade to latest HCL Foundry on-premises
  3. Post-installation Steps

Software and Hardware Requirements

Refer to the HCL Foundry Deployment Guide for the software requirements and typical deployment topologies (development and production).

Approaches to upgrade to latest HCL Foundry on-premises

  1. If you are using pre-6.5 versions of middleware or a non-Foundry version of 6.5 for on-premises, you need to install Foundry. The following are the available options to install Foundry.

    • Option 1: Installation support from the HCL Installation Services team:

      The HCL Installation Services team provides support to set up Foundry for you at a nominal charge. Refer Cost for the Services Offered for more information.

    • Option 2: Install Foundry on your own:

      1. If you are using HCL Server (middleware) but not KMS and Sync, do the following:
        1. Install Foundry using the Foundry installer. During installation, choose New Installation.
          Please follow the installation steps from the provided links:
          • Foundry Installer Guide - Windows: Installing HCL Foundry on Windows
          • Foundry Installer Guide - Linux: [Installing HCL Foundry on Linux](http://docs.hcl.com/hcllibrary/mobilefoundry/hcl_mobilefoundry_linux_install_guide/Default.htm#Installing_HCL_Foundry_on_Linux.htm?TocPath=Installation%2520Modes Installing%2520HCL%2520Foundry%2520on%2520Linux%2520-%2520Console-)

            Note: HCL recommends using the Foundry Installer for deploying Foundry 7.0 onwards. If you want to install Foundry manually due to your business rules and policies, please contact support@hcl.com for approval and to obtain access to the Foundry manual install artifacts. Manual installs not approved by HCL will not be supported.

            Note: If you are installing Foundry on AIX, contact support@hcl.com.

      2. If you are using HCL Server (middleware) with KMS or Sync, do the following:

        1. If you are using KMS pre-6.5 (non-Foundry version of KMS), do the following:

          • Raise a product support request for the DB scripts to be run to upgrade your current KMS schema to be compatible with flyway.

          • Once you receive the scripts from the product support team, run the scripts.

          • Rename the HCL Engagement Services (KMS) database to a name adhering to the format: **hclpnsdb**. For example, **hclhclpnsdb29**

        2. If you are using Sync pre-6.5 (non-Foundry version of Sync), you need to upgrade Sync manually as specified in the Upgrade of Sync Server section.

        3. Install Foundry using the Foundry installer. During installation, choose New Installation. Follow the installation steps from the provided links:

          (In case you have KMS, when the system prompts you for prefix and suffix during the installation, use the same prefix and suffix as provided earlier).

          • Foundry Installer Guide - Windows: [Installing HCL Foundry on Windows](http://docs.hcl.com/hcllibrary/mobilefoundry/hcl_mobilefoundry_windows_install_guide/Default.htm#Installing_HCL_Foundry_on_Windows.htm?TocPath=Installation%2520Modes Installing%2520HCL%2520Foundry%2520on%2520Windows _____0)
          • Foundry Installer Guide - Linux: [Installing HCL Foundry on Linux](http://docs.hcl.com/hcllibrary/mobilefoundry/hcl_mobilefoundry_linux_install_guide/Default.htm#Installing_HCL_Foundry_on_Linux.htm?TocPath=Installation%2520Modes Installing%2520HCL%2520Foundry%2520on%2520Linux%2520-%2520Console-)
        4. Then, link the upgraded Sync server with the Foundry server. Refer Linking KMS/Sync Server with Foundry for more information.
  2. If you are using pre-6.5 versions of middleware or non-Foundry version of 6.5 (On-premises pre-7.x server) and want to migrate to HCL Cloud, refer Migrating from On-premises pre-7.x Server to Cloud Foundry 7.x
  3. If you are using pre-7.x Cloud and want to migrate to 7.x Cloud, refer Migrating from pre-7.x Cloud to 7.x Cloud
  4. If you are using Foundry version of 6.5.2 middleware, do the following:
    1. Install Foundry using the Foundry installer. During installation, select Upgrade.

    2. After the upgrade is complete, republish your Foundry applications.

      Note: For KMS: If you are currently using HCL Engagement Services pre-6.5 (non-Foundry version of KMS), then during the installation do not select Engagement as you need to upgrade KMS manually as specified in the section Upgrade of HCL Engagement Server (KMS)and then link to the Foundry Server. Refer to Linking KMS/Sync Server with Foundry.

      Note: For Sync: If you are currently using Sync pre-6.5 (non-Foundry version of Sync), then during the installation please do not select Sync as you need to upgrade Sync manually as specified the section Upgrade of Sync Server, and then link to the Foundry Server. Refer to Linking KMS/Sync Server with Foundry.

Post-installation Steps

  1. If you are using custom servlets, filters, or listeners: As 7.x does not support custom servlets or filters (for example, the classes that extend HttpServlet or implement ServletContextListener or Filter), you need to modify them. Refer [How to Use Custom Servlets, Filters, and Listeners](http://docs.hcl.com/hcllibrary/mobilefoundry/hcl_mobilefoundry_user_guide/Default.htm#UsingCustomServletsFiltersAndListeners.htm?TocPath=Features Integration _____18) to make the changes and create the jar file.

    Important: The custom servlets, filters, or listeners must be complied against the latest compatible middleware-system.jar file , which can be downloaded using the Download option from Admin Console.

  2. Create the Foundry application from Iris Enterprise by following the MADP Upgrade document. The created Foundry application is displayed in the Foundry console, as follows:

    The service names of your pre-7.x application would become operation names in the Foundry application. Your existing services are grouped based on their service type and the base URL and given an auto-generated service name. If you want to see how your existing services are grouped, you can see them in Foundry Console or you can refer File <project name>_MWMappings.properties in the Iris workspace.

  3. Ensure that your client application is linked to the Foundry app.

    If your client application is linked to Foundry app properly, you would see the following UI in the Iris when you click the Project Explorer->Foundry->Use existing, where it shows the UNASSOCITATE against the Foundry app to which you have linked your client app.

  4. Copy the custom application-specific properties files (if you have any) in the middleware home.

    Note: How do I know the middleware home on my setup?

    The middleware.home is passed as the -D parameter for the server startup. For example, for Tomcat you can see it in the tomcat\conf\catalina.properties file.

  5. Please note that all the properties from the pre-7.x middleware.properties file (located in <middleware.home>\middleware\middleware-bootconfig) are now made available as the entries in the DB in Foundry v 7.x Foundry. The properties are available in the server_configuration table of the admindb database. You can configure the properties in this table.

  6. If you have scrapper services in your application, refer the Steps for Scraper services section and perform the server-side changes, if applicable.

  7. Publish the Foundry application.
  8. If you are using Sync and using custom authentication, refer the How to configure the custom authentication section.
  9. Then, build the client side application and test it.

  10. For any services issues on the server side, refer the log files.

    How can I know the path of the log file?

    The log file path is in the server_configuration table of the admindb database. The SERVER_LOG_LOCATION is the property name whose value is taken as the log file path.

    Separate log files can be seen for each of the Foundry server components, as follows:

    • middleware.log (Integration Server)
    • kms.log (Engagement Server)
    • syncservice.log (Sync Server)
    • admin.log (Admin Console)
    • authService.log (Identity Service)
    • mbaasportal.log (Foundry Workspace)
    • accountService.log (Accounts)
