AppFactory User Guide: Running your First App > Prerequisites

Prerequisites

There are a few prerequisites to complete the configuration of your build process, for your first app. Before starting, ensure you have met these requirements. You need to obtain requirements such as Project source code, key files, and credentials to complete the configuration of your first app.

The following are the prerequisites to configure an App for the first time in AppFactory Console.

VoltMX Cloud Account

VoltMX Cloud account is the most basic requirement to access AppFactory Console. You can sign up for it at the VoltMX Portal (https://manage.hcl.com).

HCL AppFactory Environment

To access AppFactory Console, the user needs access to the VoltMX AppFactory environment. The AppFactory Console and Testing Console links are available under the Environments tab at VoltMX Foundry (https://manage.hcl.com).

VoltMX Foundry Environment

VoltMX Foundry environment allows a user of AppFactory to execute, build, and publishing processes. All users of the AppFactory Cloud need to have access to a VoltMX Foundry environment. The VoltMX Admin provides necessary permissions to all users to access VoltMX Foundry environment.

GitHub 2FA

To use AppFactory with GitHub 2FA (two-factor authentication), follow these steps:

  1. Configure 2FA on GitHub. For more information, refer to Configuring two-factor authentication.
  2. Create personal access tokens. For more information, refer to Creating a personal access token for the command line.
  3. For the PROJECT_SOURCE_CODE_REPOSITORY_CREDENTIALS_ID field, click ADD.
  4. From the Kind list, select User Name with Password.
  5. Provide the Username and Password, and then ADD the credentials.
    • For the Username, type your Git username.
    • For the Password, type your Personal Access Token.
  6. Provide an ID and a Description.
  7. From the drop-down list for PROJECT_SOURCE_CODE_REPOSITORY_CREDENTIALS_ID, select the ID that you entered for the GitHub Credentials.

For more information, refer to Adding New Source Code Repository Credentials.

Iris project in source control

The developer needs to provide the link of Iris project hosted in any git supported server before the AppFactory setup. This repository should contain atleast one project or it can be in a sub-folder. Typically, the Iris source is stored as a root directory or using the project name as the document root within the git repo. The following screenshot shows an example app called MyProject and the contents of the project stored in the source control.

If you check your entire project into Git under the project directory, then the root of the project is the project name. In this case, it is MyProject.

Tests in source control

Iris project in source control requires testing. Considering this requirement, the **/test/TestNG** folder must contain Appium tests. The tests shall support Maven build management system and contains pom.xml under **/test/TestNG**. Compile the tests using **mvn compile** target.

AppFactory also gives its users the privilege to run custom tests against their Iris project’s build. If you want to test your app, then simply put the test project in **/test/TestNG** folder. It must be Appium tests. The tests support Maven build management system and contains pom.xml under **/test/TestNG**. Compile the tests using **mvn compile** target.

In the example structure, the test scripts for the project are located in MyProject/test/TestNG.

From the release V8SP3 , we have added support for Desktop Web testing. Desktop Web testing script must exist at below path MyProject/test/TestNG/DesktopWeb.

Foundry project in source control

A Foundry app is a collection of configuration files for services and their assets. As a preliminary step, push the VoltMX Foundry project to your source control repository. The repository can contain multiple Foundry apps in organized sub-directories (with unique identification).

A Foundry app directory must contain unzipped packages of the VoltMX Foundry apps. To create a package, export an app from the Foundry Console. The app is downloaded as a zip file. When you extract the zip file, it creates an export folder for the Foundry app that contains the source for the app. Make sure that you check-in the Foundry app folder to your repository.

Note:
You need to include a pom.xml file to manage the Java assets and build the projects within the AppFactory build pipeline.

You can follow different strategies to create a repository for your Foundry app. For example: Your repository can contain a single project for a Foundry app at the root, this means that your folder structure will look similar to the following screenshot.

For this repository structure, the FOUNDRY_DIR parameter must remain empty. While this is the simplest strategy for a repository, it may not be the best fit for Foundry apps that leverage custom Java code. Placing the Foundry application at the root leaves no room to add the Java code with it. Therefore, you need to manage your Java source code in a different repository.

HCL recommends a Monorepo strategy. A Monorepo for AppFactory must contain sub-directories for different Foundry Apps, and can contain a submodule for Java assets such as Preprocessors, Postprocessors, and source files for Java services. The Monorepo can also contain Maven configuration files (pom.xml) to manage the Java assets and compiling the source to generate binaries.

Note:

  • If you host the Java assets in a separate repository, you need to prefix the repository name to the Java projects path in the build parameters. For more information, refer to JAVA_PROJECTS_DIR.
  • You can also refer to a sample repository on GitHub, which contains a Foundry app and the dependent Java assets.

For more information on Monorepo strategies, refer to the following sections.

Approach 1: Common Subdirectory

In the common subdirectory approach, the Foundry App and Java assets together in the same subdirectory, but in different folder structures. Every Java asset must contain a Maven POM file to manage the Java assets and to package them into the Foundry App.

Note:
The parent java directory can also contain a common Maven POM file to customize the java packages.

Approach 2: Multiple Subdirectories

In the multiple subdirectories approach, every asset is stored in a subdirectory that contains the source file, a Maven POM file, and a gitignore file. The parent subdirectory can also contain a common Maven POM file.

Note:
If you are already using an existing structure (refer to the screenshot) of the Foundry App in Git for the AppFactory jobs, you can continue to use the same.

If you want to integrate Java assets from the source code into the Foundry app, HCL recommends that you migrate to one of the Monorepo structures that are mentioned earlier in this section. With the recommended folder structure, you can easily track and use the buildFoundryApp pipeline. You can use Git commands to retain the commit history while migrating to the new structure.

Apple Signing Certificates (Provide Apple Developer Account or Apple Manual Certs)

The developer must specify Apple Developer credentials to build an app. Depending on the type of profile for the signing app, the developer needs to generate different profiles. Check Distribution methods in Xcode help for further reference. User can provide their account credentials in the following ways:

  • Apple Developer Account Credentials:

This is the default recommended way to provide account credentials. The user need to provide their Apple Developer account username and password. AppFactory uses apple ID credentials during the build process to create a provisioning certificate and profile for building the app.

  • Apple Manual Certificates:

User can also provide Apple Manual Certificates. An app developer can use their existing apple certificates and profile (instead of sharing their apple ID credentials). To use your existing apple certificates and profile, follow these steps:

  1. Generate the proper provisioning certificate and profile from the apple developer portal.
  2. Download the certificates and manually upload to AppFactory from Build Parameters screen.

For detailed information on uploading the certificate to AppFactory, refer APPLE SIGNING CERTIFICATES.

Note: In this method, the user must re-upload the certificates if they expire.

Key store file, key store password and key password for Android

Signature data is required for APK signing. For further reference, click Sign Your App in Android Studio help.

Protected Keys

If an app developer has the protected keys and want to build their app in protected mode to enable extra security, the AppFactory user can upload their protected keys using credential plugin to use it as build parameters while building the app in Protected Mode. To know more about building an app in protected mode using protected keys, refer Protected Build.

Email address

Send email notifications after every build or test set is performed. Email address needs to be provided before the AppFactory infrastructure setup. You must provide it before the infrastructure setup to change all the configurations automatically.
